Sistem Email Routing Lensaplus

Dokumentasi cara kerja inbound, outbound transactional, dan Send-As via Gmail
Untuk tim editorial & admin · Versi April 2026 · lensaplus.com

1. Ringkasan Arsitektur

Email Lensaplus jalan di tiga jalur terpisah yang masing-masing memakai provider berbeda. Pemahaman cepat: Resend kirim notifikasi sistem ke jurnalis, Cloudflare terima email masuk ke @lensaplus.com, dan Brevo jadi pintu keluar saat tim balas dari Gmail memakai alamat @lensaplus.com.

+──────────────────────────────────────────────────────────────────+ │ JALUR 1 — OUTBOUND TRANSACTIONAL (Sistem → Jurnalis) │ │ Article workflow → Resend API → email penerima │ +──────────────────────────────────────────────────────────────────+ │ JALUR 2 — INBOUND ROUTING (Dunia → @lensaplus.com) │ │ redaksi@lensaplus.com → Cloudflare MX → forward Gmail │ +──────────────────────────────────────────────────────────────────+ │ JALUR 3 — OUTBOUND SEND-AS (Tim → Dunia, atas nama @kw) │ │ Gmail compose → Brevo SMTP → keluar dari @lensaplus.com │ +──────────────────────────────────────────────────────────────────+

Provider yang dipakai

Provider Peran Kuota gratis
Resend REST API untuk kirim email transactional (notifikasi status artikel ke jurnalis & editor) 100 email/hari
Cloudflare Email Routing Tampung MX record lensaplus.com, terapkan rule forwarding per alamat ke inbox Gmail tim Unlimited
Brevo SMTP relay SMTP keluar untuk Gmail "Send As" — supaya balasan terlihat berasal dari @lensaplus.com 300 email/hari
Gmail (akun pribadi tim) UI untuk baca & tulis. Simpan inbox, threading, search, labelling. Send-As alias diatur per akun. Standar Gmail

2. Outbound Transactional (Sistem → Jurnalis)

Library: src/lib/email.ts. Wrapper tipis di atas Resend SDK. Fungsi-fungsi yang ada:

Fungsi Dipanggil saat
sendNewReviewEmail() Jurnalis klik "Kirim untuk Review" → email ke editor yang di-assign
sendArticleApprovedEmail() Editor klik Approve di panel review
sendArticleRejectedEmail() Editor klik Tolak — sertakan catatan revisi yang harus diperbaiki
sendArticlePublishedEmail() Saat status berubah jadi PUBLISHED — baik manual oleh admin maupun via cron /api/cron/publish untuk artikel yang dijadwalkan
sendEmail() Fungsi dasar — dipakai semua wrapper di atas

Template & identitas pengirim

Semua email pakai baseTemplate() — HTML responsif dengan logo Lensaplus, accent navy #002045 / crimson #b7102a, dan tombol CTA yang membawa penerima kembali ke panel atau halaman publik.

From-address default Lensaplus <noreply@lensaplus.com> di-set lewat env EMAIL_FROM, dengan fallback ke SystemSetting.notification_email_from yang bisa diubah dari /panel/pengaturan tanpa redeploy.

Fail-silent. Kalau Resend belum dikonfigurasi (API key kosong) atau provider sedang down, sistem tidak melempar error — workflow artikel tetap jalan, tapi notifikasi email skip diam-diam. Audit log di DB tetap mencatat status change-nya, jadi tidak ada data hilang.

Yang dibutuhkan supaya jalur ini aktif

RESEND_API_KEY="re_xxxxxxxx"
EMAIL_FROM="Lensaplus <noreply@lensaplus.com>"
NEXT_PUBLIC_APP_URL="https://lensaplus.com"

Domain lensaplus.com harus diverifikasi di Resend dashboard (DNS record) supaya from-address tidak masuk spam.

3. Inbound Email (@lensaplus.com → Gmail tim)

Cloudflare Email Routing duduk di MX record domain. Saat ada email masuk ke nama@lensaplus.com, Cloudflare mencocokkan dengan rule yang sudah dibuat dan meneruskan ke email pribadi tim (biasanya Gmail).

Alur penerimaan

  1. Pengirim eksternal compose email ke redaksi@lensaplus.com.
  2. Server pengirim cek MX record → route1.mx.cloudflare.net dst.
  3. Cloudflare cek rule. Misal "redaksi@lensaplus.com → forward ke owen.lensaplus@gmail.com".
  4. Email diteruskan ke Gmail penerima. Penerima baca, label, balas seperti email biasa.
  5. Kalau penerima ingin balas atas nama @lensaplus.com bukan @gmail.com, dia pakai Send-As (Jalur 3 di bawah).

MX record yang sudah aktif

lensaplus.com  MX  10  route1.mx.cloudflare.net
lensaplus.com  MX  10  route2.mx.cloudflare.net
lensaplus.com  MX  10  route3.mx.cloudflare.net

Verifikasi destination

Saat dibuatkan email baru di panel admin atau lewat Cloudflare Dashboard, alamat Gmail penerima pertama kali harus verifikasi — Cloudflare kirim email konfirmasi, penerima harus klik link di dalamnya. Setelah itu rule baru aktif.

4. Manajemen Routing dari Panel Admin

Halaman /panel/email (akses SUPER_ADMIN saja) menyediakan UI untuk semua operasi rule routing tanpa harus buka Cloudflare Dashboard.

Yang bisa dilakukan dari panel

  • List semua email — alamat @lensaplus.com yang sudah dibuat, status aktif/ nonaktif, dan tujuan forwarding-nya.
  • Buat email baru — pilih dari user terdaftar (auto-fill local-part & destination dari profil) atau input manual localPart + destinationEmail. Validasi format Indonesia jurnalistik (huruf kecil, alfanumerik, titik, dash).
  • Toggle on/off — non-aktifkan sementara tanpa menghapus rule. Cocok saat seseorang cuti panjang.
  • Hapus permanen.
  • "Setup Gmail" — expand panduan Send-As step-by-step + tombol copy-to-clipboard untuk SMTP credential Brevo.

Endpoint API yang dipakai

Method Path Aksi
GET /api/email-routing List rules
POST /api/email-routing Buat rule baru + register destination (kirim email verifikasi)
PATCH /api/email-routing Toggle enabled
DELETE /api/email-routing?id=... Hapus rule

Endpoint diproteksi NextAuth + role check SUPER_ADMIN. Library di src/lib/cloudflare-email.ts wrap Cloudflare API v4 — semua request lewat api.cloudflare.com/client/v4/zones/{zoneId}/email/routing/....

5. Outbound Send-As (Reply dari @lensaplus.com via Gmail)

Cloudflare Email Routing tidak bisa mengirim email keluar — dia hanya untuk inbound. Untuk reply atau compose baru atas nama @lensaplus.com, Gmail user perlu konfigurasi Send-As dengan SMTP Brevo sebagai relay.

Setup di Gmail (per user, satu kali)

  1. Buka Gmail → Settings (ikon gear) → See all settings.
  2. Tab Accounts and Import → bagian Send mail as → klik Add another email address.

  3. Isi:

    • Name: nama lengkap
    • Email address: nama@lensaplus.com
    • Centang Treat as an alias

  4. Pilih Send through SMTP (bukan Gmail SMTP). Isi:

    SMTP Serversmtp-relay.brevo.com
    Port587 (TLS)
    Usernamedidapat dari Brevo dashboard, format xxxxxxxxx@smtp-brevo.com
    PasswordSMTP master password (dari Brevo dashboard, bukan password login Brevo)
  5. Gmail kirim kode konfirmasi ke nama@lensaplus.com. Karena alamat itu sudah di-forward ke Gmail user (Jalur 2), kode masuk otomatis ke inbox Gmail-nya. Masukkan kodenya → done.
  6. Setelah ini, saat compose, ada dropdown "From" — pilih nama@lensaplus.com. Email keluar lewat Brevo, headernya bersih (DKIM + DMARC sudah authenticated di domain), tidak masuk spam.
Catatan SMTP credential. Username dan password Brevo SMTP saat ini shared antar semua tim. Kalau salah satu kompromis, semua user kena. Untuk skala lebih besar, pertimbangkan Brevo paid plan dengan SMTP credential per-user, atau migrasi ke Google Workspace yang centralized.

DNS records pendukung Brevo (sudah live)

TXT     brevo-code              (verifikasi domain)
CNAME   brevo1._domainkey       (DKIM signing)
CNAME   brevo2._domainkey       (DKIM signing)
TXT     _dmarc                  v=DMARC1; p=none; rua=...

6. Test & Diagnostik

Endpoint /api/email/test

SUPER_ADMIN bisa POST ke endpoint ini untuk verify Resend config tanpa harus generate event di app: 

POST /api/email/test
{ "to": "alamat-anda@gmail.com" }

→ { "success": true, "to": "...", "from": "Lensaplus <...>" }

Yang dicek: API key valid, from-address authenticated, Resend accept payload. Kalau gagal, response berisi error dengan pesan dari Resend.

Cek logs

  • Resend dashboard → tab Logs — list semua email yang dikirim 7 hari terakhir + status (delivered/bounced/complaint).
  • Cloudflare dashboard → Email Routing → Activity — log inbound forwarding.
  • Brevo dashboard → Statistics → SMTP — log email keluar via Send-As.

7. Yang Harus Dijaga (Gotchas)

Gejala Penyebab umum Cara cek
Email notifikasi tidak sampai sama sekali RESEND_API_KEY kosong / salah POST /api/email/test — kalau success false, perbaiki env atau SystemSetting
From-address default "noreply" EMAIL_FROM tidak di-set Update env atau SystemSetting notification_email_from
Email masuk folder spam DKIM/DMARC belum authenticated Cek DNS record di Cloudflare; pastikan Brevo sudah verify domain
Forward Cloudflare tidak jalan Rule disabled atau destination belum verify Buka /panel/email → cek toggle & status verifikasi
Send-As Gmail gagal autentikasi SMTP SMTP password salah atau Brevo blok IP Login Brevo dashboard → SMTP & API → re-issue password
Hit 100 email/hari Resend Limit free plan Upgrade Resend, atau batch notifikasi sehingga 1 email = banyak event
Audit trail. Setiap perubahan status artikel yang men-trigger email otomatis tercatat di AuditLog — kolom action berisi STATUS_CHANGE dan detail menjelaskan siapa yang trigger. Kalau ada user lapor "tidak terima email", cek dulu apakah event-nya pernah terjadi (di audit log + Resend dashboard) sebelum nuduh sistem.

9. Ringkasan untuk Editorial

  1. Sistem otomatis kirim email notifikasi setiap kali artikel pindah status (submit review, approved, rejected, published). Tidak ada yang perlu diatur per artikel — Resend handle.
  2. Setiap anggota tim bisa punya alamat @lensaplus.com sendiri yang auto-forward ke Gmail mereka. Admin bikin via /panel/email.
  3. Untuk balas dari alamat @lensaplus.com (bukan dari @gmail.com), tiap anggota tim setup Send-As di Gmail-nya satu kali — pakai SMTP Brevo. Panduan ada di tombol "Setup Gmail" di panel.
  4. Semua infrastruktur (Cloudflare, Brevo, Resend) gratis untuk skala saat ini. Naik level kalau sudah lebih dari 100 notifikasi/hari atau 300 email keluar/hari.
